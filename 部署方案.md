# 技术选型.md

## 1. 项目背景

本项目为一个基于 **Next.js + Socket.IO** 的实时视频同步观看与聊天系统，需满足：

- 多用户实时视频同步（播放/暂停/跳转）
- 实时聊天
- 房间管理
- 快速部署、低成本维护
- 适用于课程设计/毕业设计场景

---

## 2. 技术选型目标

| 目标 | 说明 |
|------|------|
| **简化部署** | 避免多服务拆分带来的复杂运维 |
| **实时通信稳定** | WebSocket 长连接需保证可靠性与低延迟 |
| **数据一致性** | 前后端状态同步，无跨域问题 |
| **低成本** | 尽量使用免费或低成本云服务 |

---

## 3. 备选方案对比

| 方案 | 架构 | 优点 | 缺点 |
|------|------|------|------|
| **方案A：Vercel + 独立Socket服务** | Next.js 部署于 Vercel，Socket.IO 单独部署于其他平台（如 Render） | Vercel 对 Next.js 优化好 | 跨域、CORS、Cookie 问题；需维护两个服务；部署复杂；WebSocket 升级可能失败 |
| **方案B：Railway 一体化部署** | Next.js + API Routes + Socket.IO 共用一个 Node.js 进程，全部部署于 Railway | 单服务部署；无跨域；数据共享简单；自动 HTTPS；CI/CD 集成 | 仅支持单实例（内存存储限制）；服务重启数据丢失 |
| **方案C：自建服务器（Nginx + PM2）** | 自购 VPS，部署完整 Node 服务 | 完全可控 | 运维成本高；需手动配置 HTTPS、防火墙；无自动扩缩容 |

---

## 4. 最终选型：Railway 一体化部署

**决策理由**：

- **极简部署**：只需一个 GitHub 仓库，Railway 自动检测 Next.js 并一键部署。
- **消除跨域**：前后端同域名、同端口，无需配置 CORS。
- **内存数据共享**：API Routes 和 Socket.IO 运行在同一进程，可直接使用 JavaScript `Map` 存储房间状态，无需 Redis。
- **自动 HTTPS/WSS**：Railway 自动提供 SSL 证书，WebSocket 安全连接无需额外配置。
- **适合演示场景**：课程设计、毕业设计等对并发和数据持久性要求不高的场景，此方案最省事。

---

## 5. 架构概述

```
Internet
    │ (HTTPS / WSS)
    ▼
Railway (Node.js 进程)
    ├── Next.js App Router (页面渲染)
    ├── Next.js API Routes (业务接口)
    ├── Socket.IO Server (实时通信)
    └── 内存存储 (Map 管理房间、用户、视频状态、聊天记录)
```

- **单一进程**：所有请求（HTTP + WebSocket）进入同一服务。
- **API 与 Socket 共享内存**：`Map` 对象在模块级别保持，所有路由和事件处理器均可直接读写。

---

## 6. 数据存储选型：内存 Map

| 考虑项 | 选型 |
|--------|------|
| 数据库 | **无**，使用 `Map` 存储于内存 |
| 理由 | - 数据量小（房间数有限）<br>- 读写极快<br>- 无需额外服务<br>- 符合“仅内存存储”的设计要求 |
| 限制 | - 重启丢失所有数据<br>- 单实例不支持水平扩展 |

---

## 7. 通信协议选型：Socket.IO

| 考虑项 | 选型 |
|--------|------|
| 协议 | Socket.IO（基于 WebSocket，降级轮询） |
| 理由 | - 成熟的实时通信库<br>- 自动重连、心跳机制<br>- 与 Next.js 集成良好<br>- 广播和房间支持开箱即用 |
| 替代方案 | 原生 WebSocket（但需手动处理重连、事件封装等） |


## 9. 适用场景与未来扩展

- **当前适用**：课程设计、毕业设计、小型团队内部演示、MVP 原型。
- **未来扩展建议**：
  - 如需持久化 → 接入 Redis 存储房间状态。
  - 如需多实例 → 使用 Redis 共享存储 + 消息发布/订阅（如 Socket.IO Redis Adapter）。

---

## 10. 结论

**Railway 一体化部署** 是本项目最优的技术选型，它在简化部署、开发效率和实时通信稳定性之间取得了最佳平衡，完全满足毕业设计的要求，并为后续升级预留了清晰的扩展路径。
