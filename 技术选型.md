# 技术选型

## 1. 项目背景

本项目为一个基于 **Next.js + Socket.IO** 的实时视频同步观看与聊天系统（SyncTube / 同步影院），需满足：

- 多用户实时视频同步（播放/暂停/跳转）
- 实时文字聊天
- 房间管理（创建/加入/在线用户列表）
- 快速部署、低成本维护
- 适用于课程设计/毕业设计场景

---

## 2. 选型目标

| 目标 | 说明 |
|------|------|
| 简化部署 | 避免多服务拆分带来的复杂运维，单一仓库一键部署 |
| 实时通信稳定 | WebSocket 长连接需保证可靠性与低延迟 |
| 数据一致性 | 前后端状态同步，无跨域问题 |
| 低成本 | 尽量使用免费或低成本云服务 |
| 开发效率 | 类型安全、组件复用、逻辑分层清晰 |

---

## 3. 前端框架：Next.js 15 (App Router)

| 考虑项 | 选型 |
|--------|------|
| 框架 | Next.js 15.1.3 (App Router) |
| 理由 | 文件路由系统、SSR/SSG 支持、API Routes 内置、Vercel/Railway 原生支持；App Router 提供 layout、loading、error boundary 等内置能力 |
| React 版本 | React 19.0.0 |
| 替代方案 | Vite + React Router（缺少 API Routes 和 SSR）、Nuxt 3（生态偏向 Vue） |

---

## 4. 语言：TypeScript (strict)

| 考虑项 | 选型 |
|--------|------|
| 语言 | TypeScript 5.7.2，strict 模式 |
| 理由 | 类型安全减少运行时错误；Socket 事件映射通过 `ServerToClientEvents` / `ClientToServerEvents` 联合类型保证前后端 payload 一致；IDE 智能提示提升开发效率 |
| 类型定义 | `types/index.ts` 统一定义 `Room`、`OnlineUser`、`ChatMessage`、`VideoState` 及所有 Socket 事件 payload |

---

## 5. 样式方案：Tailwind CSS v3 + shadcn/ui 风格

| 考虑项 | 选型 |
|--------|------|
| CSS 框架 | Tailwind CSS 3.4.17 |
| 理由 | 原子类快速开发、暗色主题友好、无运行时开销、与 Next.js 零配置集成 |
| 组件原语 | shadcn/ui 风格（非 npm 安装，源码内复制） |
| 理由 | 完全可控源码、无版本锁定、按需裁剪；已实现 `Button`、`Input`、`Card`、`ScrollArea`、`Toast`、`Badge` 六个原语 |
| 字体 | Geist（Vercel 开源字体，通过 `geist` npm 包引入） |
| 设计系统 | 暗色玻璃拟态（glassmorphism）：`backdrop-blur-sm` + 半透明深色背景 + `border-white/5` 细边框；单一霓虹蓝强调色 `#00d4ff`；圆角规范 `xl`/`2xl`/`full` |

---

## 6. 动画：Framer Motion

| 考虑项 | 选型 |
|--------|------|
| 动画库 | framer-motion 11.15.0 |
| 理由 | 声明式 API（`initial`/`animate`/`transition`）、支持 `prefers-reduced-motion` 无障碍特性、与 React 19 兼容 |
| 用途 | 主页标题入场、房间加载动画、消息淡入、悬停/点击微交互 |
| 替代方案 | CSS animation/transitions（复杂动画维护成本高）、GSAP（过重） |

---

## 7. 图标：Phosphor Icons

| 考查项 | 选型 |
|--------|------|
| 图标库 | @phosphor-icons/react 2.1.7 |
| 理由 | 树摇友好（按需导入）、多 weight（bold/fill/regular）、风格统一、与暗色主题搭配协调 |
| 替代方案 | lucide-react（图标数量较少）、react-icons（多库混合，体积大） |

---

## 8. 视频播放：react-player + Bilibili iframe 降级

| 考虑项 | 选型 |
|--------|------|
| 播放器 | react-player 2.16.0 |
| 理由 | 统一接口支持 MP4、HLS、YouTube、Vimeo 等多平台；提供 `seekTo`、`onProgress`、`onDuration` 等命令式 API，与同步逻辑配合良好 |
| B站处理 | `lib/video.ts` 识别 `bilibili.com/video/BVxxx` 和 `AVxxx` 链接，生成 `player.bilibili.com/player.html` 可嵌入 URL |
| 限制 | B站 iframe 无法程序化控制播放/暂停/进度，仅能保证两端加载同一视频；`b23.tv` 短链接无法直接内嵌（需服务端解析重定向） |
| 替代方案 | video.js（配置复杂）、原生 `<video>`（不支持 YouTube/HLS） |

---

## 9. 实时通信：Socket.IO

| 考虑项 | 选型 |
|--------|------|
| 协议 | Socket.IO（基于 WebSocket，降级轮询） |
| 理由 | 成熟的实时通信库；自动重连、心跳机制；内置 Room 概念天然适配房间广播；与 Next.js 集成良好 |
| 替代方案 | 原生 WebSocket（需手动处理重连、事件封装、房间管理）、Pusher/Ably（第三方 SaaS，有免费额度限制） |

### 当前状态：Mock Socket 层

项目当前使用 `lib/mockSocket.ts` 作为 Socket.IO 客户端的 mock 实现，通过 `BroadcastChannel` 在同一浏览器的不同标签页间广播事件。接口完全对齐 `socket.io-client`（`on`/`off`/`emit`/`connect`/`disconnect`/`id`），迁移到真实 Socket.IO 只需将 `getSocket()` 替换为 `io(serverURL)`。

### 参考服务端

`server/index.ts` 提供了基于 Socket.IO 的参考服务端实现，处理 `room:join`、`chat:send`、`video:state`、`video:seek` 等事件，使用内存 `Map` 管理房间状态。该文件已从 `tsconfig.json` 编译范围排除（依赖未安装的 `socket.io` 包），供后续集成时参考。

---

## 10. 数据存储：内存 Map

| 考虑项 | 选型 |
|--------|------|
| 数据库 | 无，使用 JavaScript `Map` 存储于内存 |
| 理由 | 数据量小（房间数有限）；读写极快；无需额外服务；符合项目文档"仅内存存储"的设计要求 |
| 数据结构 | `Room { roomId, users: Map<socketId, OnlineUser>, videoState: VideoState, messages: ChatMessage[] }` |
| 限制 | 服务重启丢失所有数据；单实例不支持水平扩展；聊天消息上限 200 条防止内存溢出 |
| 未来扩展 | 如需持久化可接入 Redis；如需多实例可使用 Socket.IO Redis Adapter |

---

## 11. 部署平台：Railway 一体化部署

### 备选方案对比

| 方案 | 架构 | 优点 | 缺点 |
|------|------|------|------|
| Vercel + 独立 Socket 服务 | Next.js 部署于 Vercel，Socket.IO 单独部署于 Render 等 | Vercel 对 Next.js 优化好 | 跨域 CORS 问题；需维护两个服务；WebSocket 升级可能失败；部署复杂 |
| Railway 一体化部署 | Next.js + API Routes + Socket.IO 共用一个 Node.js 进程 | 单服务部署；无跨域；内存共享；自动 HTTPS/WSS | 仅支持单实例；服务重启数据丢失 |
| 自建服务器（Nginx + PM2） | 自购 VPS 部署完整 Node 服务 | 完全可控 | 运维成本高；需手动配置 HTTPS、防火墙 |

### 最终选型：Railway

**决策理由**：

- 极简部署：只需一个 GitHub 仓库，Railway 自动检测 Next.js 并一键部署。
- 消除跨域：前后端同域名同端口，无需配置 CORS。
- 内存数据共享：API Routes 和 Socket.IO 运行在同一进程，`Map` 对象在模块级别保持，所有路由和事件处理器可直接读写。
- 自动 HTTPS/WSS：Railway 自动提供 SSL 证书，WebSocket 安全连接无需额外配置。
- 适合演示场景：课程设计、毕业设计等对并发和数据持久性要求不高的场景，此方案最省事。

### 架构概述

```
Internet
    | (HTTPS / WSS)
    v
Railway (Node.js 进程)
    +-- Next.js App Router (页面渲染)
    +-- Next.js API Routes (房间创建/加入/查询 REST 接口)
    +-- Socket.IO Server (实时通信: 播放同步/聊天/用户列表)
    +-- 内存存储 (Map 管理房间、用户、视频状态、聊天记录)
```

- 单一进程：所有请求（HTTP + WebSocket）进入同一服务。
- API 与 Socket 共享内存：`Map` 对象在模块级别保持，所有路由和事件处理器均可直接读写。

## 12. 部署前置条件

当前项目使用 mock Socket 层，部署前需完成以下集成：

1. 安装 `socket.io` 和 `socket.io-client` 依赖。
2. 将 Socket.IO Server 集成到 Next.js 自定义服务端（custom server），监听 HTTP 请求和 WebSocket 连接。
3. 将 `lib/mockSocket.ts` 中的 `getSocket()` 替换为 `io(serverURL)`，或保留 mock 作为开发环境降级。
4. 实现 `app/api/rooms/route.ts`（房间创建/加入/查询 REST 接口）。
5. 配置 Railway 部署（连接 GitHub 仓库，自动构建）。

## 14. 适用场景与未来扩展

- 当前适用：课程设计、毕业设计、小型团队内部演示、MVP 原型。
- 未来扩展建议：
  - 如需持久化：接入 Redis 存储房间状态。
  - 如需多实例：使用 Redis 共享存储 + 消息发布/订阅（Socket.IO Redis Adapter）。
  - 如需语音聊天：接入 WebRTC（PeerJS 或自建信令服务器）。
  - 如需 B 站 1080P：通过服务端代理 B 站登录态，获取高清晰度播放权限。
